# feat: v1.1 Seal Content Protection

## Summary

Encrypt the L1 `Model3D` base GLB when its `LicenseTerms.policy` is ALLOW_LIST or RESTRICTED (PERMISSIONLESS stays public), gating decryption with a Move `seal_approve` that checks the soulbound `NftCollectionCreatorCap` (ALLOW_LIST) or `caller == creator` (RESTRICTED). Encryption is derived from policy and fixed at publish. Ships as a **fresh package republish**. Forkers decrypt client-side, then derive variants via the existing backend bake (no forker-facing raw download). Preview = client-side Babylon stills for ALLOW_LIST only. Post-6/21 v1.1.

---

## Problem Frame

Today every L1 base GLB is public on Walrus and the `is_encrypted` flag is decorative (only ever `false`, never checked; frontend hardcodes it at `frontend/src/creator/CreateModelPage.tsx:641`). The on-chain derive fee (`launch_collection`) is therefore bypassable: scrape the base, fork off-platform, never pay. Creators who want to charge for forking (ALLOW_LIST) or keep a base private (RESTRICTED) have a lever that does not hold. This plan makes the lever real with Seal. See origin: `docs/brainstorms/2026-05-30-v1.1-seal-content-protection-requirements.md`.

## System-Wide Impact

- **Move package** `model3d::model3d` — shipped as a **fresh republish** (new package ID) with new `seal_approve*` entry fns, `is_encrypted` derived from policy, an amended `launch_collection` policy gate split into cap-issuing + token-minting steps, an `ALLOW_LIST ⇒ fee > 0` assert, and new `Model3D` struct fields (sealed key, preview blob ids, seal version). Existing public models stay on the old package, untouched.
- **Frontend** — new Seal client module; ALLOW_LIST re-exposed in the publish UI; encrypted publish path; the 3-step forker decrypt path; client-side preview capture; catalog rendering that excludes RESTRICTED and shows previews for ALLOW_LIST; landing guard-test updates.
- **Backend** — same material-swap behavior, with added hardening for encrypted-base requests (no body logging, no persisted plaintext, JWT wallet-address check).
- **Docs/decisions** — amends D-040 (launch_collection gate + re-enabling ALLOW_LIST in the UI), new ADRs (Seal dependency, envelope encryption, fresh-republish deployment), rewrites `spec.md §3.7`, resolves OQ-026.

## High-Level Technical Design

*Directional guidance for review, not implementation specification. The implementing agent should treat it as context, not code to reproduce.*

**Envelope encryption** (avoids Seal-encrypting a multi-MB mesh, which would bloat bytes against the Walrus 35/46 MB encoder cliff — Seal's documented pattern):

```
PUBLISH (encrypted policy):
  aesKey = random 256-bit                         (client)
  ciphertextGLB = AES-256-GCM(baseGLB, aesKey)    (client)  → ~same size as plaintext
  sealedKey = Seal.encrypt(aesKey, id, threshold) (client)  → small
      where id = [model_id_bytes][nonce]   (Seal namespaces the packageId itself; id is the inner bytes only)
  glb_blob_id      ← upload ciphertextGLB to Walrus (existing standalone blob path)
  sealed_key       ← struct field on the (republished) Model3D
  seal_version     ← struct field (the package version at publish time)
  preview_blob_ids ← (ALLOW_LIST only) client-side Babylon stills, public Walrus blob(s), struct field

DECRYPT + FORK (forker, ALLOW_LIST) — 3 steps, NOT one atomic tx:
  (1) launch_collection(model, payment≥fee)  → soulbound NftCollectionCreatorCap   [cap, no tokens yet]
  (2) SessionKey.create + one personal-message signature
      txBytes = PTB calling seal_approve_cap(id, cap, collection, model)
      aesKey  = Seal.decrypt(sealed_key, sessionKey, txBytes)   (key servers dry-run seal_approve)
      baseGLB = AES-GCM-decrypt(fetch(ciphertextGLB), aesKey)   (client)
      variants = backend bake(baseGLB)  → upload quilt
  (3) mint_tokens(collection, quilt_blob_id, token_patch_ids)
  PERMISSIONLESS keeps the existing one-popup atomic launch_collection_with_tokens.
```

**`seal_approve` shape** (two overloads, side-effect-free, dry-run by key servers; `id` prefix-bound to the model id; `seal_version` checked):

```move
// ALLOW_LIST: holder of the soulbound cap linked to a collection derived from this model.
// Triple-check is a NAMED INVARIANT — all three asserts required, each isolation-tested.
entry fun seal_approve_cap(id, cap: &NftCollectionCreatorCap, collection: &NftCollection, model: &Model3D)
  // assert: cap.collection_id == id(collection)         (cap authorizes THIS collection)
  //       && collection.base_model_id == id(model)      (collection derives from THIS model)
  //       && id starts_with id(model)                   (ciphertext id bound to THIS model)
  //       && model.seal_version == VERSION              (gate not relaxed by a later upgrade)

// RESTRICTED: only the recorded creator
entry fun seal_approve_creator(id, model: &Model3D, ctx)
  // assert: id starts_with id(model) && ctx.sender() == model.creator && model.seal_version == VERSION
```

## Key Technical Decisions

- **Deployment: fresh package republish, not a compatible upgrade** — per D-040's precedent, an enforcement change (new gate, encryption) must republish: a compatible upgrade leaves the old, ungated bytecode callable at the old package ID. Republishing also *simplifies* storage — a new package adds fields directly to the `Model3D` struct, so no dynamic-field workaround is needed. Existing public models remain on the old package (no migration; consistent with "all existing models are public").
- **Envelope encryption (AES-256-GCM + Seal-wrapped key)** rather than Seal-encrypting the whole GLB — keeps the Walrus blob ~plaintext-sized (clear of the 35/46 MB OOM cliff; mesh decimation was declined) and is Seal's documented pattern for large payloads.
- **New `Model3D` struct fields** (in the republished package): `sealed_key`, `preview_blob_ids`, `seal_version`. The encrypted GLB stays in the existing `glb_blob_id` (now holding AES-ciphertext when encrypted); `is_encrypted` tells the client which read path to take.
- **`seal_approve_cap` triple-check is a named invariant** — `cap.collection_id == id(collection)` ∧ `collection.base_model_id == id(model)` ∧ `id` prefix — each asserted and isolation-tested (one failing check per test). Prevents a valid cap from unlocking another model's ciphertext (the canonical Seal binding pitfall). RESTRICTED → creator-equality.
- **`launch_collection` splits into cap-issuing + token-minting** for encrypted ALLOW_LIST forks — the cap is needed to decrypt, but decryption must precede the variant bake that the token mint consumes, so the atomic one-tx launch cannot stand. Step 1 mints the cap (fee paid), steps 2-3 decrypt/bake/mint. PERMISSIONLESS retains the atomic path.
- **ALLOW_LIST requires `derivative_mint_fee > 0`** (on-chain assert at publish) — with no on-chain address allowlist in v1, ALLOW_LIST's only meaning is "pay to fork"; `fee = 0` would let anyone get a cap free and decrypt, making it logically identical to PERMISSIONLESS-with-pointless-encryption. The "free fork, earn via royalty" strategy belongs in PERMISSIONLESS (royalty applies there too). *(Caveat: revisit this assert if a future curated-address allowlist makes free-but-gated forking meaningful.)*
- **`is_encrypted` derived from policy in `new_model`** (encrypted ⇔ policy ≠ PERMISSIONLESS), fixed at publish — closes the decorative-flag gap.
- **RESTRICTED is private** — excluded from the public catalog/browse entirely, and generates no preview (creator-only; no external evaluator, per origin R11). Previews are an ALLOW_LIST-only affordance.
- **`seal_version` binding** — the package version is stored at publish and asserted in `seal_approve`, so a future upgrade of the new package cannot silently relax the gate over already-encrypted models. (Low cost; the version check rides the key-server dry-run, which charges no gas.)
- **Backend bake retained, with hardening** (user decision, 2026-05-30) — the decrypted base transits our JWT-authed backend over TLS for material-swap exactly as public bases do today; the forker gets no download button. Refines origin R9: "no raw download" = no forker-facing download, not "plaintext never leaves the browser." For encrypted-base requests the backend must not log the request body, must not persist plaintext, and must verify the submitting JWT's wallet holds the in-flight cap.
- **Testnet key servers: 2-of-3 Mysten independent (Open mode), threshold 2** — robust to one server outage, no API key. Mainnet (by 8/27) needs a contracted provider; deferred.

## Implementation Units

### U1. Seal client module + dependency

**Goal:** Add `@mysten/seal` and a frontend Seal wrapper providing envelope encrypt/decrypt and SessionKey handling.
**Requirements:** R3, R5, R7, R8 (enables).
**Dependencies:** none.
**Files:** `frontend/package.json`, `frontend/src/seal/sealClient.ts` (client + key-server config + SessionKey TTL constant), `frontend/src/seal/envelope.ts`, `frontend/src/seal/sessionKey.ts`, `frontend/src/seal/sealClient.test.ts`, `frontend/src/seal/envelope.test.ts`.
**Approach:** Pin `@mysten/seal@1.1.3` (verify against `@mysten/sui@2.16.2`, 2026-05-08 train). `serverConfigs` = the two Mysten testnet object IDs (threshold 2; structure for a third). `encryptBase(bytes, model_id) → { ciphertext, sealedKey, idHex }` does AES-256-GCM with a random key then `client.encrypt` of the 32-byte key with `id = [model_id_bytes][nonce]` (inner bytes only — Seal adds the package prefix). `decryptKey(sealedKey, sessionKey, txBytes) → aesKey` then `decryptBase(ciphertext, aesKey)`. SessionKey helper creates + caches a per-package session behind an explicit TTL constant (default ~2-3 min for the short forge flow) and exposes the personal-message to sign.
**Patterns to follow:** `frontend/src/walrus/walrusClient.ts` for client-construction style; existing `signPersonalMessage` usage in `useSession`.
**Test scenarios:**
- Envelope round-trips for small + multi-MB GLB buffers.
- AES ciphertext length ≈ plaintext + GCM overhead (guards the OOM-cliff assumption — but see U3 verification for the end-to-end encoder re-validation).
- Tampered ciphertext / wrong key → GCM auth failure throws.
- `id` is unique per call (nonce) and carries the model-id prefix; `id` passed to encrypt is inner bytes only (no package prefix).
- SessionKey caches within the TTL constant; expiry forces a fresh signature.
**Verification:** Unit tests green; `pnpm typecheck` clean; no Seal calls on the PERMISSIONLESS path.

## Risk Analysis & Mitigation

- **Walrus OOM cliff (35/46 MB).** AES-GCM ciphertext ≈ plaintext size, so encryption barely moves the needle. The U1 byte-parity test guards raw size, but the cliff is the *encoder's* memory — **re-validate with a real large encrypted publish end-to-end** (U3 verification) rather than relying on the byte test alone.
- **`id`↔object binding error** lets a cap unlock the wrong ciphertext. Mitigation: the named triple-check invariant + isolation tests in U2.
- **Fork-flow ordering** — resolved by the 3-step split (cap → decrypt/bake → mint). Risk: the split is a larger Move surface; U2 tests both the cap-only and mint steps.
- **D-040 amendment** is a real behavior change (ALLOW_LIST becomes non-creator-forkable, and re-appears in the UI). Mitigation: explicit ADR, U2/U3 tests for allow + still-deny cases.
- **fee = 0 footgun** — closed by the on-chain `ALLOW_LIST ⇒ fee > 0` assert.
- **SessionKey UX** adds a personal-message signature; wallet-gated, can't be driven in agent-browser. Mitigation: split verification per the protocol; explicit short TTL; consider the Enoki signer path.
- **Testnet key servers have no SLA / key persistence** — surface to demo evaluators so a decrypt failure isn't read as a product bug; do not encrypt anything that must be recoverable long-term.
