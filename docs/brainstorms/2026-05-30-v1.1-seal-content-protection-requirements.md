# v1.1 Seal Content Protection

## Summary

Turn the currently-decorative `is_encrypted` flag into real Seal encryption for non-public bases. ALLOW_LIST bases are encrypted so only forkers who paid the derive fee can decrypt and derive **in-app**; RESTRICTED bases are encrypted for creator-only use; PERMISSIONLESS stays public. Encryption is derived from the base's policy and fixed at publish. Built post-6/21.

---

## Problem Frame

Today every L1 `Model3D` base GLB is public on Walrus. The on-chain fork/derive fee (`launch_collection`) that pays the base creator is therefore only enforced by convenience: anyone can scrape the base off the public aggregator, fork it off-platform, and never pay — and the `is_encrypted` flag that was meant to gate this is never checked (it only ever asserts `false` in tests). For creators who want to charge for forking (or keep a base private), the platform currently offers a lever that silently doesn't hold. This is anticipatory — no live user has reported it yet; it is reasoned from the bypass being structurally trivial while the base is public.

Separately, `spec.md §3.7` still describes a Seal design against the deleted `Access` struct (see OQ-026); that stale design must be replaced by this one before any Seal code lands.

---

## Actors

- A1. **Mesh creator** — publishes a `Model3D` base and chooses its `LicenseTerms.policy`, which determines whether the base is encrypted.
- A2. **NFT creator / forker** — evaluates a base from its preview, pays the derive fee to fork an ALLOW_LIST base, then decrypts and builds variants in-app.
- A3. **Buyer** — purchases public L2 `NftToken`s; unaffected by Seal (listed only to mark the boundary: L2 stays public).
- A4. **Seal key servers** — the t-of-n threshold operators that dry-run the on-chain policy and release decryption key shares.

---

## Key Flows

- F1. **Encrypted publish**
  - **Trigger:** A1 publishes a base under ALLOW_LIST or RESTRICTED.
  - **Actors:** A1
  - **Steps:** Client generates public preview asset(s) from the plaintext base → encrypts the base GLB → uploads the encrypted base + public preview + public lineage in one upload.
  - **Outcome:** Catalog shows the base as a public preview; the usable mesh is ciphertext on the public read path.
  - **Covered by:** R1, R3, R4, R5, R12, R13

- F2. **Forker unlock & derive (ALLOW_LIST)**
  - **Trigger:** A2 wants to fork an encrypted ALLOW_LIST base.
  - **Actors:** A2, A4
  - **Steps:** A2 views the public preview → pays the derive fee via `launch_collection` → receives the soulbound creator cap → key servers see the cap and release shares → base decrypts in the Forge → A2 derives variants and publishes them publicly.
  - **Outcome:** A2 has a collection; the base creator was paid; no raw base file was handed over.
  - **Escape path:** A wallet without the cap is denied key shares; the base stays encrypted, preview still viewable.
  - **Covered by:** R6, R7, R8, R9

- F3. **Exclusive base (RESTRICTED)**
  - **Trigger:** A1 publishes a base only they may use.
  - **Actors:** A1, A4
  - **Steps:** Base encrypted at publish → only A1's wallet satisfies the policy → A1 decrypts and launches their own collections; all other wallets denied.
  - **Outcome:** No third-party forks; A1 monetizes via their own collections.
  - **Covered by:** R10, R11

## Requirements

**Policy-derived encryption**
- R1. Encryption is determined by `LicenseTerms.policy`, not an independent toggle: PERMISSIONLESS → unencrypted; ALLOW_LIST and RESTRICTED → encrypted. `is_encrypted` becomes a derived consequence of policy, set once.
- R2. Policy (and therefore encryption state) is fixed at publish and immutable thereafter; there is no post-publish transition between policies.

**Encrypted publish**
- R3. For an ALLOW_LIST or RESTRICTED base, only the base GLB is encrypted before upload; the lineage record and preview asset(s) stay public.
- R4. An encrypted publish produces the encrypted base and its public preview together in a single upload, adding no wallet approval beyond today's publish flow.
- R5. Encrypted base bytes are served from the existing public aggregator + CDN unchanged; protection comes from encryption (the key gate), not from gating storage or the CDN.

**Forker access — ALLOW_LIST**
- R6. Before paying, a prospective forker can view a public preview sufficient to decide whether to fork; the full mesh is never delivered pre-payment.
- R7. Paying the base's derive fee (the existing `launch_collection` path) is the sole gate: it yields the soulbound creator cap, and holding that cap is what authorizes decryption of that base.
- R8. A wallet that has not paid the derive fee for a given base cannot decrypt it.
- R9. After unlocking, the forker derives variants from the decrypted base **in-app only** — the product offers no raw base-GLB download. Derived L2 variants publish publicly as today.

**Exclusive base — RESTRICTED**
- R10. A RESTRICTED base is decryptable only by its creator; no third party can fork or decrypt it.
- R11. RESTRICTED requires no preview asset (no external evaluator).

**Preview**
- R12. Preview assets are generated client-side at publish from the plaintext base (no server-side rendering). Default treatment is the full look; reduced treatments (clay / part-reveal) are optional creator choices that do not affect the gate.
- R13. Preview assets are public and carry a visible watermark.

**Framing**
- R14. The feature is positioned as mitigation, not prevention: it blocks unauthorized access to the base and explicitly does not claim to stop an authorized forker from extracting the mesh after decryption.

---

## Acceptance Examples

- AE1. **Covers R6, R8.** Given an ALLOW_LIST base, when an un-paid wallet requests it, the public preview is visible but key-share release is denied and the mesh stays encrypted.
- AE2. **Covers R7, R9.** Given a wallet that paid the derive fee (holds the cap), when it opens the base in the Forge, the base decrypts in-app and variants can be built and published — with no raw-download option presented.
- AE3. **Covers R10.** Given a RESTRICTED base, when any wallet other than the creator attempts to decrypt, release is denied; the creator's own wallet succeeds.
- AE4. **Covers R2.** Given a base published as PERMISSIONLESS, when the creator later wants it gated, there is no conversion path — it stays public.

---

## Success Criteria

- A creator who chooses ALLOW_LIST/RESTRICTED can confirm their base GLB is not freely downloadable from the public aggregator without paying; casual scraping is blocked.
- A paid forker's in-app decrypt-and-derive works end-to-end; a non-payer is cleanly denied while the preview stays visible.
- `ce-plan` can implement without inventing product behavior: the gate (the soulbound cap), what is encrypted (base GLB only), the immutability rule, the derive-in-app constraint, and the preview source (client-side) are all specified here.

---

## Scope Boundaries

- Forensic / per-buyer watermark for leak tracing — separate feature (OQ-008).
- L2 `NftToken` / variant gating — L2 stays public by design (community building on collections).
- Transferable, time-limited, or subscription access — would need a struct beyond the soulbound cap; v1.2+.
- Per-variant gating inside the L2 quilt (one blob co-locates patches; cannot gate within it).
- The 6/21 deny→allow demo slice — out of this doc by the chosen post-6/21 full-feature scope.
- Changing fee values or defaults — fee weighting stays the creator's call.
- Defeating a determined authorized forker who re-uploads the decrypted base, and the related seam where a public L2 variant reveals base topology — accepted limits (mitigate, not prevent), bounded by royalty being the hard on-chain rail.

---

## Key Decisions

- **Encryption derived from policy, not a separate toggle** — kills the pointless combo (PERMISSIONLESS + encrypted) and the leaky combo (RESTRICTED/ALLOW_LIST + unencrypted = the original decorative-flag bug).
- **Gate on the existing soulbound creator cap, no new struct** — the cap is `key`-only / soulbound and is obtained only by paying the derive fee, so it already is a soulbound paid-access receipt.
- **Immutable policy at publish** — a public base cannot be un-leaked, so post-publish gating would be meaningless; choosing RESTRICTED-as-exclusive (not draft) confirms this.
- **Only the base GLB is encrypted; lineage stays public** — preserves on-chain/Walrus provenance.
- **Derive-in-app only** — maximizes leak friction and matches how the Forge already builds variants.
- **Ciphertext on the existing public CDN** — Seal gates the key, not the bytes, so the D-073 read path is untouched.
- **Royalty is the hard rail; fork fee is soft** — the platform provides the optional Seal lever via policy; fee weighting stays the creator's decision.

## Dependencies / Assumptions

- `@mysten/seal` SDK + available Seal key servers (testnet first; mainnet only if the deploy timeline reaches it).
- `spec.md §3.7` is stale (OQ-026) and must be rewritten onto this design as part of implementation; an ADR should capture the redesign.
- Demand is anticipatory — reasoned from fork-fee bypass, not an observed user request.
- Existing published models are all PERMISSIONLESS/public; no migration is required.
- The L2-topology-leak seam is accepted as a known limit.
- Related decisions: D-029/D-030 (Access deleted), D-032 (`Model3D` shared), D-073 (read-path CDN), D-004 (royalty cap). Seed: `docs/ideation/2026-05-30-content-protection-seal-ideation.md`.
